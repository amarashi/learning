The List, Decoded

The ~60 names you found, dropped onto Lesson 01's map. The point isn't to learn them all — it's to see the shape.

Three things make this list stop being scary: (1) it's only ~6 categories; (2) inside most categories the items are interchangeable alternatives — you pick one; (3) over half is back end, which your goal (read & tweak the front end) doesn't require — and you already know its friendliest language, Python.

FRONT END — runs in the browser your zone

This whole half is your mission. You already run the modern default stack here: React + TypeScript + Vite + npm.

③ The three browser languages

HTML✓ UsePage structure. The bedrock — non-negotiable.
CSS✓ UseStyling. The bedrock.
JavaScript✓ UseThe only language the browser runs. Everything compiles down to this.

① UI frameworks — pick one (they all do the same job)

React✓ UseBuild UI from reusable components. The market leader.
Next.jsA framework built on React — adds routing, server-rendering, etc. Reach for it when a React site needs pages, SEO, or a bit of backend.
Vue.js · Angular · SvelteDirect alternatives to React. Different syntax, same purpose. You don't need them if you're on React.

① Styling helpers — pick one (optional)

Tailwind CSSWrite styling as tiny utility classes instead of separate CSS files. Very popular now. (Your FHA site uses plain CSS + design tokens instead.)
SassCSS with extras (variables, nesting); compiles to plain CSS. Older convenience layer.
BootstrapReady-made styled components (buttons, grids). Older; fast but generic-looking.

① Helper libraries

TypeScript✓ UseJavaScript + type-checking. Spans front end and back end.
AxiosA small library to fetch data from a server. (The browser's built-in fetch now does most of this.)

② Build & quality toolchain

npm✓ UseInstalls & tracks libraries. Your pip.
Vite✓ UseThe build tool / dev server. The modern default.
Webpack · Esbuild · TurbopackOther bundlers. Webpack = the old standard; esbuild = the fast engine Vite uses inside; Turbopack = Vercel's newer one. You're on Vite — just recognize the others.
ESLintLinter — flags likely bugs & bad patterns as you type.
PrettierAuto-formatter — reformats code consistently on save. Cosmetic, not logic.

CROSS-CUTTING — both halves use these recognize

② Version control & code hosting

Git✓ UseTracks every change to your code (undo, history, branches). Universal.
GitHub · GitLabWebsites that host Git repos + collaboration. GitHub is the default; GitLab is the main alternative.

Hosting / cloud / infrastructure — where the site lives online

Vercel · NetlifySimple hosts that deploy a front-end site straight from GitHub. Easiest path to "live on the internet."
AWS · GCP · AzureThe big cloud providers (Amazon / Google / Microsoft). Rent servers, databases, storage. Powerful, complex, enterprise.
DockerPackages an app + its environment into a portable "container" that runs identically anywhere. Like a reproducible env for a whole app.
KubernetesRuns & scales many Docker containers across many machines. Heavy infrastructure — far from your goal.

Testing — code that checks your code

Jest · VitestRun small "unit" tests on functions. Vitest is the Vite-native one.
Playwright · CypressDrive a real browser to test whole user flows ("click login, expect dashboard"). End-to-end testing.

BACK END — runs on a server just recognize

Out of scope for your goal. Your FHA site has no back end. You only need to recognize these as "the server/database half." Bonus: you already know Python — the gentlest entry here — so if a project ever needs a back end, you're closer than you think.

Server frameworks — build the API/server

Node.jsRuns JavaScript outside the browser — it's what powers npm & Vite on your machine, and most JS back ends.
Express · NestJSBack-end frameworks for Node (JS/TS). Express = minimal; Nest = structured.
Django · FastAPIPython back-end frameworks. You'd be at home here — FastAPI is the modern choice for serving APIs (incl. ML models).
Spring Boot · .NET · Ruby on Rails · LaravelBack-end frameworks for Java / C# / Ruby / PHP. Enterprise & legacy worlds. Name-only.

General-purpose languages (mostly back-end here)

Python✓ KnowYou already speak this. Common back end + the language of your ML day job.
Java · C# · Go · Rust · PHPOther back-end languages. Go & Rust = modern/fast; Java & C# = enterprise; PHP = legacy web. Recognize, don't study.

Databases — where data is stored

PostgreSQL · MySQL · SQLiteRelational (table-based, SQL). Postgres = modern default; SQLite = a single file, zero setup.
MongoDBDocument database (flexible, JSON-like). "NoSQL" — no fixed table schema.
RedisIn-memory store — blazing fast, used for caching & queues.

APIs — how front talks to back

REST APIThe standard convention for a front end to request data over HTTP. A pattern, not a tool.
GraphQLAn alternative to REST — the front end asks for exactly the fields it wants.
PostmanA desktop tool to poke at an API by hand while developing. Not part of the app.

Backend-as-a-service — skip writing a back end

Firebase · SupabaseReady-made back ends: database + login + hosting in one. Great when a front-end dev needs data/auth without building a server. Supabase = open-source, Postgres-based.

So what should you actually touch?

Mapped against your goal — read & tweak the front end:

  • Already yours (the core): HTML, CSS, JS, React, TypeScript, npm, Vite, Git. We deepen these.
  • Worth a glance soon: Tailwind (you'll see it everywhere), GitHub, Vercel/Netlify (how a site goes live), ESLint/Prettier.
  • Recognize the category, ignore the rest: all back end, all databases, AWS/Docker/Kubernetes, the testing tools. They'll still make sense — because you know which bucket they're in.
