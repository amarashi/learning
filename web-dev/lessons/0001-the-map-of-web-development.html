Lesson 01 · The Big Picture

The Map of Web Development

Before any tool makes sense, you need the territory. This is the one map the whole roadmap hangs off — and every box on it is already sitting in your FHA project.

Here is the single idea that demystifies almost everything: a web browser only speaks three languages. Everything else in web development exists either to make those three nicer to write, or to translate your nicer code back down into them.

① What you (and the AI) actually write

React build the UI from reusable pieces LandingPage.tsx · Button.tsx
TypeScript JavaScript + safety checks .ts / .tsx · tsconfig.json
Libraries someone else's code you borrow three.js · lenis

a build step translates & bundles it

② The toolchain (translator + librarian)

npm downloads & tracks libraries package.json · node_modules/
Vite build tool + live dev server vite.config.ts · npm run dev

produces plain files the browser can read

③ What the browser actually runs — the only 3 languages

HTML structure — the nouns on the page index.html
CSS style — colour, layout, type global.css · Button.css
JavaScript behaviour — what happens when you click (compiled from your .tsx)

Read it bottom-up and it tells the true story of the web. Read it top-down and it tells the story of your workflow: you write in layer ①, a tool in layer ② translates it, and the browser runs the result in layer ③.

Layer ③ — the bedrock: HTML, CSS, JavaScript

This is the floor everything stands on. A browser knows these three and nothing else. The classic way to remember the split:

  • HTML is the structure — headings, paragraphs, buttons, images. The nouns. Your index.html is the page's skeleton.
  • CSS is the style — what those nouns look like. Your Navy/Pink/Azure colours and the Montserrat font live in styles/global.css.
  • JavaScript is the behaviour — what happens over time and on interaction (open a menu, play the hero video, smooth-scroll).

Reference: MDN's three intro modules — HTML, CSS, JavaScript.

Layer ① — why we don't write the bedrock directly

You could hand-write thousands of lines of raw HTML/CSS/JS. Nobody does, for the same reason you don't write assembly to run a regression. We write in something more expressive and let a tool compile it down.

  • React is a library for building a UI out of reusable components — little functions that return a chunk of page. Your Button.tsx is defined once and reused everywhere. The HTML-looking syntax inside those files is JSX; the build step turns it into plain JavaScript. (react.dev/learn)
  • TypeScript is JavaScript plus a type checker — it catches "you passed a string where a number goes" before the code ever runs. That's the .ts/.tsx and tsconfig.json in your repo.
  • Libraries are pre-built code you borrow instead of writing. three draws the 3-D hero; lenis does the buttery scroll. You didn't build those — you installed them.
Maps to what you already know You never write machine code — you write Python or R, and an interpreter runs it. The web is the same shape: HTML/CSS/JS is the machine layer; React/TypeScript is your high-level language; and the build tool is the compiler in between. Knowing this, most of the roadmap stops being a pile of unrelated words and becomes "another option inside one of these three boxes."

Layer ② — the toolchain: npm and Vite

Two jobs sit between what you write and what ships:

  • npm = the package manager. package.json is your project's recipe (which libraries, which versions); node_modules/ is where the downloaded library code lands.
  • Vite = the build tool. In development, npm run dev starts a live server that recompiles the instant you save. For release, npm run build bundles everything into plain optimised files in dist/. (Why Vite)
Maps to what you already know npm install is pip install / install.packages(), and node_modules/ is your virtual-env / library folder. npm run dev is a live notebook that re-runs on save — same loop as re-running a cell when the plot looks wrong.

One honest gap: the "back end"

This whole map is the front end — everything that runs in the visitor's browser. Many apps also have a back end: a server + database that store and serve data. Your FHA site is a marketing site, so it's front-end only — there's no back end to find in the repo. We'll name that part of the map later, when a project actually needs it.

Your win this lesson You can now take any file or buzzword in a web project and drop it into one of three buckets — browser language, author convenience, or toolchain — and say why. That's the skeleton every future lesson hangs on.
