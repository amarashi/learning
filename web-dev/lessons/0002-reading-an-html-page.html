Lesson 02 · Layer ③ — HTML

Reading an HTML Page

"Read & tweak" starts here. Last time you got the map; now we land in the bedrock box — and read your FHA project's actual index.html, line by line.

HTML has essentially one repeating shape. Learn to see that one shape and you can read any page on the web — including the surprising, nearly-empty one sitting at the root of your own project.

The one shape: an element

Everything in HTML is an element. An element is a piece of content wrapped in a matching pair of tags. Here is the whole grammar in one specimen:

<a href="/about">Our team</a>

opening & closing tag attribute name attribute value content (what you see)
  • The tags mark where the element starts and ends. The closing tag is the same name with a slash: <a> … </a>. The name (a) decides what kind of thing it is — here, a link.
  • Attributes live inside the opening tag and configure the element: href="/about" says where the link goes. They're always name="value" pairs.
  • The content is what sits between the tags — often the visible text, or more elements.
Maps to what you already know An element is just a function call with the contents as the body and attributes as keyword arguments — think a(href="/about", "Our team") in R/Python. And because content can hold more elements, a page is a nested tree, exactly like nested lists or a JSON object. Indentation in the file shows the nesting depth.

A handful of tags do not wrap content — <meta> and <img> are self-closing (no separate end tag) because they are their attributes. That's the only exception to the one shape.

Reference: MDN — Anatomy of an HTML element & basic HTML syntax.

Every page has the same skeleton

Three elements wrap every HTML document, and they nest in a fixed order:

<html>            the whole document
  <head>          info about the page — invisible (title, settings, links to CSS)
  <body>          the page itself — everything the visitor actually sees
</html>

So the first question to ask of any HTML file is: is this line in the head (machine-facing) or the body (human-facing)? That single split explains most of what you'll read.

Now read your own: FHA's index.html

This is the real, unedited file from your project. Read it top to bottom — you now know every shape in it:

<!doctype html>                          "this is a modern HTML file"
<html lang="en">
  <head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1" />
    <meta name="description" content="Future Health Australia — …accelerating data, digital and AI-enabled care." />
    <title>Future Health Australia — Bridging innovation and impact</title>
  </head>
  <body>
    <div id="root"></div>
    <script type="module" src="/src/main.tsx"></script>
  </body>
</html>

your file · fha/index.html

The <head> is exactly what we said — machine-facing settings: the language, the character set, the mobile viewport rule, the search-engine description, and the browser-tab title. Nothing here is drawn on the page.

The surprise in the body

Look at the highlighted lines. The entire visible part of your website is two lines, and the first one is an empty box:

  • <div id="root"></div> — a div is a generic container (a plain box with no meaning of its own). This one is empty. The id="root" is just a name tag so other code can find it.
  • <script … src="/src/main.tsx"> — this loads and runs your JavaScript application. It finds the empty root box and fills it with the entire page — the hero, the nav, every section.
This is the big one In a React project, index.html is not where the page lives — it's a near-empty launchpad. The real structure is authored up in Layer ① as .tsx components, and the build step wires it to this root div. So if you go looking for a heading's text here and can't find it — that's expected. It's in a .tsx file. Knowing where NOT to look is half of reading code.
Maps to what you already know Think of index.html as a tiny main() that does almost nothing except call your real program. root is the empty plot device you hand to your library; main.tsx is the script that draws into it. The page you see is rendered output, not the source.
Your win this lesson You can open any HTML file and name what you're looking at — element, tag, attribute, content — tell head (machine) from body (human), and explain why a React app's index.html is nearly empty. You just read your own project's front door.

Quick check — recall only, no clues in the wording:

1. In <a href="/about">Team</a>, what is href="/about"?

2. Your <body> is basically <div id="root"></div>. Why is the page nearly empty?

3. What does <script type="module" src="/src/main.tsx"> do on this page?
